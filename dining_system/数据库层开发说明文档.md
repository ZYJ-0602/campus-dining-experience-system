# 校园食堂匿名评价系统 - 数据库层开发说明文档

## 1. 概述
本文档旨在说明本系统数据库层的设计思路、核心表结构、关键业务逻辑及异常处理机制，适用于毕业设计答辩及系统维护。本系统采用 SQLite 作为轻量级数据库，通过 SQLAlchemy ORM 框架进行操作。

## 2. 表结构说明

本系统共包含 5 张核心数据表，均已通过 SQLAlchemy 模型类定义并自动生成。

### 2.1 用户表 (`user`)
*   **用途**: 记录匿名评价者的设备信息，用于防刷单和行为追踪。
*   **字段**:
    *   `id` (Integer, PK): 自增主键。
    *   `device_id` (String, Unique): 设备唯一标识符（如 device_123456），非空。
    *   `create_time` (DateTime): 首次访问时间。

### 2.2 窗口表 (`window`)
*   **用途**: 存储食堂窗口的基础信息，是评价系统的核心主体。
*   **字段**:
    *   `id` (Integer, PK): 自增主键，对应前端二维码中的 window_id。
    *   `name` (String): 窗口名称（如“1号窗口”），非空。
    *   `canteen_name` (String): 所属食堂（如“北区食堂”），非空。
    *   `create_time`, `update_time`: 记录创建与更新时间。

### 2.3 菜品表 (`dish`)
*   **用途**: 记录各窗口的菜品信息，供用户在食品评价模块中选择。
*   **字段**:
    *   `id` (Integer, PK): 自增主键。
    *   `name` (String): 菜品名称，非空。
    *   `window_id` (Integer, FK): 外键关联 `window.id`，非空。
    *   `price` (Numeric): 价格。
    *   `is_valid` (SmallInteger): 状态标识（1=有效，0=下架），默认1。
*   **关联**: 多对一关联 `window` 表。

### 2.4 评价核心表 (`evaluation`)
*   **用途**: 存储用户提交的所有维度评价数据，是数据分析的基础。
*   **字段**:
    *   `id` (Integer, PK): 自增主键。
    *   `user_id`, `window_id` (FK): 关联用户和窗口，非空。
    *   `dish_id` (FK): 关联菜品，可空（若未选菜品）。
    *   **四维评分**: `food_score`, `service_score`, `environment_score`, `food_safety_score` (Numeric)，非空。
    *   **服务类型**: `service_type` (String)，记录打菜/刷卡人员类型，非空。
    *   **详细数据**: 各维度的标签、文字描述、图片路径等（String），可空。
    *   `is_submitted` (SmallInteger): 提交状态（0=仅保存草稿，1=正式提交）。
    *   `submit_time` (DateTime): 提交时间。

### 2.5 食安公示表 (`food_safety`)
*   **用途**: 展示窗口的食品经营许可证等合规信息。
*   **字段**:
    *   `id` (Integer, PK): 自增主键。
    *   `window_id` (FK): 关联窗口，非空。
    *   `certificate_name`, `certificate_no`: 证书名称与编号，非空。
    *   `valid_start`, `valid_end`: 有效期范围，非空。
    *   `upload_path`: 证书图片路径。

## 3. 核心数据操作逻辑

### 3.1 评价提交流程
1.  **用户识别**: 根据 `device_id` 查询 `user` 表，若不存在则自动创建新用户。
2.  **数据校验**: 检查 `window_id` 是否有效，必填评分项是否大于 0。
3.  **防重检查**: 若为正式提交 (`is_submitted=1`)，则触发防重逻辑。
4.  **数据入库**: 将各维度评分、标签、图片路径写入 `evaluation` 表。

### 3.2 防重复评价机制
*   **规则**: 同一设备 (`device_id`) 在 5 分钟内不能对同一窗口 (`window_id`) 进行重复的正式提交。
*   **实现**: 查询 `evaluation` 表中满足以下条件的记录：
    *   `user_id` 匹配当前用户
    *   `window_id` 匹配当前窗口
    *   `is_submitted = 1`
    *   `submit_time` 在 (当前时间 - 5分钟) 之后
*   **结果**: 若存在记录，则拦截提交并返回提示。

### 3.3 菜品查询逻辑
*   **规则**: 仅展示当前窗口且状态为“有效”的菜品。
*   **实现**: `Dish.query.filter_by(window_id=wid, is_valid=1).all()`。

## 4. 异常场景处理

| 场景 | 异常类型 | 处理方案 |
| :--- | :--- | :--- |
| **外键不存在** | `IntegrityError` | 提交评价前先查询 `Window.query.get(id)`，若不存在则直接返回错误提示，避免数据库级报错。 |
| **重复提交** | 业务逻辑拦截 | 在插入数据前先执行 `check_repeat_evaluation`，若返回 True 则中断流程。 |
| **数据缺失** | `TypeError/ValueError` | 接口层对必填字段（如分数、服务类型）进行非空校验，缺失则返回 400 状态码。 |
