# 毕设演示优化与答辩指南

本文档旨在帮助您在毕设答辩中流畅演示「校园食堂用餐体验点评系统」，并提供界面美化、问题修复及答辩讲解的全方位指导。

---

## 一、核心演示流程 (5-8步)

### 阶段 1：C端用户体验 (模拟学生)
1.  **登录与首页**
    *   **操作**：打开 `login.html`，输入学生账号登录 -> 跳转 `index.html`。
    *   **展示**：瀑布流笔记、智能推荐卡片、顶部搜索。
    *   **讲解**：“系统首页采用了移动端优先的瀑布流布局，集成智能推荐算法，为用户提供个性化内容。”

2.  **扫码评价**
    *   **操作**：点击首页“扫码评价” -> `evaluation_scan.html` -> 提交四维评分。
    *   **展示**：滑块交互、标签选择、提交成功提示。
    *   **讲解**：“这是核心功能之一，支持扫码直达窗口，采用了防重复提交机制（5分钟限制）和多维度量化评分。”

3.  **发布笔记**
    *   **操作**：点击首页悬浮“+”按钮 -> `post_publish.html` -> 上传图片/填写内容 -> 发布。
    *   **展示**：图片拖拽排序、级联选择器（食堂-窗口-菜品）。
    *   **讲解**：“支持图文混排和级联数据关联，前端实现了图片预览和草稿箱功能。”

### 阶段 2：B端运营管理 (模拟食堂经理)
4.  **菜品评价分析**
    *   **操作**：登录运营账号 -> `dish_evaluation_admin.html`。
    *   **展示**：查看“番茄炒蛋”的评分饼图、筛选差评、导出报表。
    *   **讲解**：“通过ECharts可视化展示菜品口碑分布，帮助食堂精准优化菜品。”

### 阶段 3：系统管理 (模拟管理员)
5.  **内容审核**
    *   **操作**：登录管理员账号 -> `admin_audit.html` -> 驳回一条违规笔记。
    *   **展示**：审核列表实时计数、敏感词库管理。
    *   **讲解**：“系统引入了双重风控机制：自动敏感词拦截 + 人工审核，确保社区内容安全。”

6.  **系统监控**
    *   **操作**：进入 `admin_index.html` (仪表盘) -> 查看今日数据。
    *   **展示**：数据大屏、实时日志。
    *   **讲解**：“管理员可实时监控系统运行状态，掌握全校食堂的舆情动态。”

## 二、界面美化建议

### 1. 视觉统一
*   **主色调**：坚持使用橙色 (`#FF7F24`) 作为主色，辅以白色背景和浅灰边框。
*   **字体**：统一使用系统默认无衬线字体 (`-apple-system, BlinkMacSystemFont...`)，保证跨平台清晰度。
*   **圆角**：卡片、按钮统一使用 `8px` 圆角 (`border-radius: 8px`)，营造亲和力。

### 2. 交互动画 (CSS)
给关键按钮添加点击反馈：
```css
/* 按钮点击缩放效果 */
.btn:active {
    transform: scale(0.95);
    transition: transform 0.1s;
}

/* 点赞红心动画 */
@keyframes heart-beat {
    0% { transform: scale(1); }
    50% { transform: scale(1.3); }
    100% { transform: scale(1); }
}
.like-btn.active i {
    color: #dc3545;
    animation: heart-beat 0.3s ease-in-out;
}
```

### 3. 图表优化 (ECharts)
配置项优化建议：
*   **配色**：使用柔和的调色盘 `['#FF7F24', '#5470c6', '#91cc75', '#fac858']`。
*   **字体**：设置 `textStyle: { fontFamily: 'sans-serif' }`。
*   **提示框**：开启 `tooltip: { trigger: 'axis', backgroundColor: 'rgba(255,255,255,0.9)' }`。

## 三、常见问题修复方案

### 1. 页面跳转失败 (404)
*   **现象**：点击按钮提示“File not found”。
*   **原因**：文件移动后相对路径未更新。
*   **修复**：使用 `organize_project.py` 脚本自动修复，或手动检查 `href` 属性。
    *   同级跳转：`href="target.html"`
    *   跨级跳转：`href="../c-client/target.html"`

### 2. ECharts 图表不显示
*   **现象**：图表区域空白。
*   **原因**：容器未设置高度或 JS 报错。
*   **修复**：
    1.  确保 CSS 中容器有明确高度：`height: 300px;`。
    2.  检查是否引入 `echarts.min.js`。
    3.  初始化代码需放在 DOM 加载后：`document.addEventListener('DOMContentLoaded', ...)`。

### 3. 模拟数据加载失败
*   **现象**：列表为空或显示 `undefined`。
*   **修复**：
    1.  确认 `mock_data.js` 是否在 HTML 底部引入。
    2.  检查 `MockLoader.init()` 是否自动执行。
    3.  控制台查看是否有报错，确认 URL 参数解析正确。

## 四、答辩讲解要点 (高分话术)

### 1. 数据库设计 (关联关系)
*   **要点**：“我的数据库设计遵循第三范式，核心是 **用户-评价-食堂** 的三角关系。”
*   **话术**：“为了支持精细化管理，我将食堂层级设计为 `Canteen -> Window -> Dish` 三级结构，评价表 `Evaluation` 同时关联用户、窗口和菜品，实现了数据来源的可追溯性。”

### 2. 核心功能实现 (防重复评价)
*   **要点**：Redis/数据库校验 + 前端限制。
*   **话术**：“为了保证评价真实性，系统实现了防刷机制。后端通过校验 `(user_id, window_id, date)` 的唯一性索引，前端则通过 `localStorage` 记录提交时间戳，5分钟内禁止重复提交，双重保障数据质量。”

### 3. 前端架构 (响应式 + 模块化)
*   **要点**：Bootstrap5 Grid + 通用组件封装。
*   **话术**：“前端采用了移动端优先的响应式设计，一套代码适配手机和PC。我封装了 `common.js` 处理通用的 API 请求和表单校验，大大降低了代码冗余度。”
